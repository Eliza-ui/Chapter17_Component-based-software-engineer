Chapter 17: Component-Based Software Engineering

CBSE Processes Overview

CBSE processes are software processes that support component-based software engineering. They take into account the possibilities of reuse and the different process activities involved in developing and using reusable componentss.

Figure 17.6 (Kotonya, 2003) presents an overview of the processes in CBSE. At the highest level, there are two types of CBSE processes:

  1. Development for reuse — concerned with developing components or services that will be reused in other applications. It usually involves generalizing existing components.
  2. Development with reuse — the process of developing new applications using existing components and services.

Process Objectives

These processes have different objectives and therefore include different activities. In the development for reuse process, the objective is to produce one or more reusable components.

You know the components that you will be working with and you have access to their source code to generalize them. In development with reuse, you don’t know what components are available, so you need to discover these components and design your system to make the most effective use of them. You may not have access to the component source code.

Component Acquisition

Component acquisition is the process of acquiring components for reuse or development into a reusable component. It may involve accessing locally developed components or services or finding these components from an external source.

1️⃣ Component acquisition — ensures software engineers can identify, evaluate, and obtain reusable components either from within the organization or third-party vendors, making it easier to integrate them into new systems.

Component Management

Component management is concerned with managing a company’s reusable components, ensuring that they are properly cataloged, stored, and made available for reuse.

2️⃣ Component management — maintains an organized repository of reusable components. It ensures they are documented, version-controlled, and accessible for developers to use in future projects.

CBSE for Reuse

CBSE for reuse is the process of developing reusable components and making them available for reuse through a component management system.

The vision of early CBSE supporters was that a thriving component marketplace would develop, with specialist component providers and vendors organizing the sale of components. However, this vision has not been realized yet.

Organizational Reuse

CBSE for reuse is most likely to take place within an organization committed to reuse-driven software engineering. They wish to exploit software assets from different parts of the company.

However, internally developed components are not usually reusable without change. They often include application-specific features that limit reuse in other systems.

Adapting Components

To make components reusable, you must adapt and extend them to create more generic versions. This adaptation has associated costs, so cost-effectiveness must be evaluated.

You must decide if a component is likely to be reused and whether the savings justify the costs of making it reusable. Stable domain abstractions, like “accounts” or “patients,” are good reuse candidates.

Improving Reusability

Changes made to improve reusability include removing application-specific methods, renaming functions, and adding more general functionality.

  • Remove app-specific methods
  • Change names for general use
  • Add methods for full coverage
  • Ensure consistent exception handling
  • Add configuration interfaces
  • Integrate and generalize interfaces

Exception Handling

Exception handling is a particularly difficult aspect. Components should not handle exceptions themselves, as each application has different requirements.

Instead, the component should define what exceptions can arise and publish these as part of the interface. For example, a stack component should detect and publish stack overflow and underflow exceptions.
